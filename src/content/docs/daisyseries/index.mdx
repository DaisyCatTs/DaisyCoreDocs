# DaisySeries

<div class="home-hero">
  <span class="home-kicker">The Lightweight Utility Toolkit</span>
  <h2>Modern Paper-first parsing for plugins that need clean values without adopting a full runtime framework.</h2>
  <p>DaisySeries turns repeated enum-helper glue into one small consistent contract: parse input, fail cleanly, store canonical keys, and show friendly names in UI.</p>

  <div class="home-actions">
    <a class="home-button" href="/daisyseries/getting-started/install/">Install DaisySeries</a>
    <a class="home-button-secondary" href="/daisyseries/getting-started/first-parse/">Parse the First Values</a>
  </div>
</div>

<div class="value-strip">
  <div class="value-pill"><strong>Config-safe</strong><span>Accept practical input, return stable keys, and remove plugin-local parser boilerplate.</span></div>
  <div class="value-pill"><strong>Tiny contract</strong><span>Every module follows the same parse, parseOrNull, key, displayName, and aliases pattern.</span></div>
  <div class="value-pill"><strong>Standalone or paired</strong><span>Use DaisySeries alone, or pair it with DaisyCore when your plugin also needs runtime systems.</span></div>
</div>

<div class="doc-shell">
  <div class="doc-lead">
    <span class="eyebrow">What it removes</span>
    <h3>No more local <code>valueOf(...)</code> wrappers, alias maps, or friendly-name helpers.</h3>
    <p>DaisySeries exists to standardize the boring but important parsing layer that otherwise gets reimplemented across every Paper plugin repo.</p>
  </div>
  <div class="doc-rail">

## What DaisySeries includes

<div class="feature-matrix">
  <div class="feature-card"><strong>Materials</strong><p>Config-friendly parsing, canonical material keys, and clean display names for UI and logs.</p></div>
  <div class="feature-card"><strong>Sounds</strong><p>Modern Paper-first sound parsing from lowercase, spaced, kebab-case, or canonical input.</p></div>
  <div class="feature-card"><strong>Item Flags</strong><p>Single-value parsing plus list parsing for item config blocks.</p></div>
  <div class="feature-card"><strong>Enchantments</strong><p>Live-registry-backed runtime parsing with canonical suggestions and stable key rules.</p></div>
  <div class="feature-card"><strong>Potions</strong><p>Potion-effect parsing centered on `PotionEffectType`, not brewing-oriented item concepts.</p></div>
</div>
