# Choose a Product

## Choose DaisyCore when you need a platform

Pick DaisyCore if your plugin needs:

- command registration and execution
- menu sessions
- sidebars
- tablists
- MiniMessage-first text and placeholder rendering
- one runtime owner through `DaisyPlatform`

Start here: [DaisyCore install](/daisycore/getting-started/install/)

## Choose DaisySeries when you need lightweight helpers

Pick DaisySeries if your plugin mainly needs:

- enum parsing from config-safe strings
- canonical lowercase serialization keys
- friendly display names
- curated aliases for real plugin inputs

Start here: [DaisySeries install](/daisyseries/getting-started/install/)

## Choose DaisyConfig when your config layer is the problem

Pick DaisyConfig if your plugin mainly needs:

- typed YAML mapping
- reload-safe config state
- validation with useful path errors
- DaisySeries-backed config codecs
- DaisyCore-friendly config-backed text sources when you want reloadable or shared wording

Start here: [DaisyConfig install](/daisyconfig/getting-started/install/)

## Use all three together when your plugin is runtime-heavy and config-heavy

This is the strongest pairing when:

- DaisyCore owns commands, menus, sidebars, or tablists
- DaisySeries parses materials, sounds, or item flags from config
- DaisyConfig owns the typed YAML and reload-safe state around those systems

Next step: [Use DaisyConfig with DaisyCore](/daisyconfig/guides/use-daisyconfig-with-daisycore/)

If you want to see the whole stack before deciding, read [Full Ecosystem Example](/overview/full-ecosystem-example/).
