# Roadmap

## Current products

- **DaisyCore**: the runtime/platform product
- **DaisySeries**: the parser/reference utility product
- **DaisyConfig**: the typed YAML, managed lifecycle, and module bundle product

## Current ecosystem priority

The current priority is ecosystem polish:

- MiniMessage-first defaults
- stronger production hardening
- clearer docs
- stronger cross-product examples
- better reference coverage
- one obvious end-to-end Daisy stack path

That work comes before the next new Daisy library.

## DaisyCore

DaisyCore currently focuses on:

- platform bootstrap
- commands
- menus
- shared text and placeholders
- items
- sidebars
- tablists

See the platform docs: [DaisyCore Overview](/daisycore/)

## DaisySeries

DaisySeries currently ships:

- materials
- sounds
- item flags
- enchantments
- potions
- biomes
- villager professions
- attributes
- entity types
- game modes
- difficulties
- block faces
- damage causes
- operations
- pattern types
- particles
- statistics

DaisySeries growth is now intentionally gated. New parser families should only land when repeated modern plugin use clearly justifies them.

See the utility docs: [DaisySeries Overview](/daisyseries/)

## DaisyConfig

DaisyConfig currently ships:

- typed YAML mapping
- managed YAML lifecycle
- module bundles
- DaisySeries-backed codecs
- DaisyCore-friendly text bridging

See the config docs: [DaisyConfig Overview](/daisyconfig/)

## Next major project after the current polish pass

The current recommendation is:

1. finish the suite integration and polish pass
2. then choose the next large library based on the strongest repeated problem

The leading candidates after this polish phase are:

- **DaisyData** for persistence, repositories, caches, and player data lifecycle
- **DaisyEffects** for particles, sounds, titles, action bars, and display-effect composition

The gate before starting either one is simple:

- the public roadmap is current
- the three shipped products have one coherent story
- one canonical full-stack guide exists
- install/version guidance is internally consistent
- migration paths are practical for real plugin authors
