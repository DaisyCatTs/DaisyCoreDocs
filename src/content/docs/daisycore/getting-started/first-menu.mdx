# First Menu

This page shows the smallest useful menu flow: open a menu inline for a player, then understand what DaisyCore turns that into at runtime.

## What this page assumes

<div class="premium-callout note">

- you already have a working DaisyCore bootstrap
- you are comfortable with the command flow from [First Command](/daisycore/getting-started/first-command/)
</div>

## What you'll build

<div class="doc-section">
You will create a simple profile menu that opens from your `/profile` command.
</div>

## Step 1: open a menu inline

```kotlin
player.openMenu("Profile", rows = 3) {
    background(Material.GRAY_STAINED_GLASS_PANE) {
        name(" ")
    }

    slot(13) {
        item(Material.PLAYER_HEAD) {
            nameLang("menus.profile.card.name", viewer = player)
            loreLang("menus.profile.card.lore", viewer = player)
        }

        messageLang("messages.profile.clicked", "player" to player.name)
        closeOnClick()
    }
}
```

Inline opening is the main ergonomic path for quick plugin UI flows.

## Step 2: open it from a command

```kotlin
command("profile") {
    player {
        player.openMenu("Profile", rows = 3) {
            slot(13) {
                item(Material.PLAYER_HEAD) {
                    nameLang("menus.profile.card.name", viewer = player)
                }
            }
        }
    }
}
```

## What a menu session is

If you want reuse, keep a `DaisyMenu` value around. When a player opens it, DaisyCore creates a `DaisyMenuSession` for that player. That session holds runtime state like rendering and refresh lifecycle.

For quick flows, inline opening is usually enough. DaisyCore still handles the session ownership under the hood.

<div class="premium-callout current">
  <h3>Current behavior</h3>
  <p>Menu definitions stay reusable while runtime state stays session-specific. That separation is one of the main reasons DaisyCore menus stay clean in plugin code.</p>
</div>

## What to keep in mind

- menu definitions are meant to stay reusable
- per-player runtime state lives in sessions
- DaisyCore cleans up sessions when the platform shuts down
- shared lang/config text can be installed once through `DaisyPlatform`
