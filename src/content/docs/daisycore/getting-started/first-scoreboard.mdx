# First Scoreboard

This page shows the modern DaisyCore sidebar path: keyed lines, session ownership, and text that can be plain, MiniMessage, or config-backed.

## What this page assumes

<div class="premium-callout note">

- your plugin has a `DaisyPlatform`
- you enabled `scoreboards()`
- you want a simple sidebar for one player-facing feature
</div>

## Step 1: start with plain strings or MiniMessage strings

```kotlin
fun profileSidebar(player: Player) =
    sidebar {
        title("<gradient:#7dd3fc:#c4b5fd>Profile</gradient>")
        line("name") {
            text("<white>${player.name}</white>")
        }
        line("coins") {
            text("<gray>Coins: <white>1,250</white>")
        }
    }
```

## Step 2: show it to a player

```kotlin
val session = daisy.scoreboards?.show(player, profileSidebar(player))
```

## Why keys matter

Each line has a key like `name` or `coins`. DaisyCore uses those keys so a session can refresh specific lines instead of rebuilding everything blindly.

This is why `line(key) { ... }` is the normal path instead of relying on anonymous lines only.

## Step 3: refresh the parts that changed

```kotlin
session?.invalidate("coins")
session?.refreshNow()
```

Use `invalidate("coins")` when one keyed line changed. Use `refreshNow()` when you want a full recompute immediately.

## Step 4: switch to shared config/lang text only when it helps

```kotlin
fun profileSidebar(player: Player) =
    sidebar {
        titleLang("sidebar.profile.title", viewer = player)
        line("coins") {
            textLang("sidebar.profile.coins", viewer = player, "coins" to "1,250")
        }
    }
```

Again, `sidebar.profile.title` is not a Daisy-only concept. It is just a config key path if your plugin stores text outside code.

## What the current implementation focuses on

- title rendering
- keyed lines
- plain strings, MiniMessage strings, or config-backed lang text
- blank line support
- targeted invalidation
- safe cleanup when the player quits or the platform closes
