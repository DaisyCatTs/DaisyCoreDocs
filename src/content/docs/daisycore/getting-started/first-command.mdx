# First Command

This page shows the normal DaisyCore command workflow: define a command group, annotate it, and let `DaisyPlatform` discover it during startup.

## What this page assumes

- you already completed [Install DaisyCore](/daisycore/getting-started/install/)
- your plugin already creates a `DaisyPlatform` like the one shown in [First Plugin](/daisycore/getting-started/first-plugin/)

## Start simple

You do not need `replyLang(...)` for basic messages.

Use:

- `reply("...")` for plain strings
- `replyMm("...")` for MiniMessage strings
- `replyLang("...")` when the text lives in config or lang files

## Step 1: create a command group

```kotlin
package cat.daisy.example.command

import cat.daisy.command.DaisyCommandGroup
import cat.daisy.command.DaisyCommandSet

@DaisyCommandSet
object ProfileCommands : DaisyCommandGroup({
    command("profile") {
        description("Open your profile tools")

        player {
            reply("Opening your profile.")
        }
    }
})
```

## Step 2: add MiniMessage when you want formatting

```kotlin
command("profile") {
    description("Open your profile tools")

    player {
        replyMm("<gradient:#7dd3fc:#c4b5fd>Opening your profile.</gradient>")
    }
}
```

## Step 3: use config-backed text when your plugin already stores messages there

If your plugin has a config or lang file, that is when `replyLang(...)` starts paying off.

```kotlin
command("profile") {
    description("Open your profile tools")

    player {
        replyLang("messages.profile.opening", "player" to player.name)
    }
}
```

The key itself is just your plugin's own config path. DaisyCore is not forcing a special naming system. `messages.profile.opening` and `tablist.profile.header` are only examples of keys a config-driven plugin might already use.

## Step 4: add a subcommand

```kotlin
command("profile") {
    description("Open your profile tools")

    subcommand("open") {
        player {
            reply("Profile opened.")
        }
    }
}
```

`command(...)`, `subcommand(...)`, and `player { ... }` stay short because they are DSL verbs. The long-lived public types stay Daisy-branded so library ownership is still obvious.

## Step 5: keep async replies clean

```kotlin
subcommand("sync") {
    player {
        loading()

        profileService.sync(player.uniqueId).replyResult(
            context = this,
            success = {
                replyMm("<green>Profile synced for <white>${player.name}</white>.</green>")
            },
            failure = { error ->
                failLang("messages.profile.error", "error" to error.message.orEmpty())
            },
        )
    }
}
```

## How discovery works

When `commands()` is enabled in your `DaisyPlatform` builder, DaisyCore scans your plugin jar for classes annotated with `@DaisyCommandSet`. Valid `DaisyCommandGroup` and `DaisyCommandProvider` implementations are loaded automatically.

That means you do not manually register this provider in normal usage.
