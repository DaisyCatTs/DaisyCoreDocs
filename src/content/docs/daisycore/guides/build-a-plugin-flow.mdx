# Build a Plugin Flow

This guide shows how the main DaisyCore systems fit together in one believable feature instead of isolated toy snippets.

<div class="premium-callout note">
  <h3>What this page assumes</h3>
  <ul>
    <li>You already have a working DaisyCore bootstrap.</li>
    <li>You have read the basics of commands, menus, scoreboards, and tablists.</li>
  </ul>
</div>

## What you'll build

A `/profile` flow:

- `/profile` opens a menu
- the menu acts as the player's profile hub
- a sidebar mirrors a few profile stats
- a tablist shows a simple welcome header and footer

## Step 1: bootstrap DaisyCore once

```kotlin
class ProfilePlugin : JavaPlugin() {
    lateinit var daisy: DaisyPlatform
        private set

    override fun onEnable() {
        daisy =
            DaisyPlatform.create(this) {
                messages(ProfileText)
                commands()
                menus()
                scoreboards()
                tablists()
            }
    }

    override fun onDisable() {
        daisy.close()
    }
}
```

## Step 2: define one shared text source

```kotlin
object ProfileText : DaisyTextSource {
    override fun text(key: String): String? =
        when (key) {
            "menus.profile.title" -> "<#f38ba8><bold>Profile</bold></#f38ba8>"
            "menus.profile.card.name" -> "<#f38ba8><bold>{player}</bold></#f38ba8>"
            "sidebar.profile.title" -> "<#f38ba8><bold>Profile</bold></#f38ba8>"
            "sidebar.profile.coins" -> "<white>Coins: <#f38ba8>{coins}</#f38ba8>"
            "tablist.profile.header" -> "<#f38ba8><bold>Welcome back</bold></#f38ba8>\n<white>{player}</white>"
            "tablist.profile.footer" -> "<gray>Coins: <white>{coins}</white>"
            else -> null
        }

    override fun textList(key: String): List<String> =
        when (key) {
            "menus.profile.card.lore" -> listOf("<gray>Inline Daisy menu flow", "<white>Built for {player}")
            else -> emptyList()
        }
}
```

## Step 3: define the sidebar and tablist

```kotlin
fun profileSidebar(player: Player) =
    sidebar {
        titleLang("sidebar.profile.title", viewer = player)
        line("coins") {
            textLang("sidebar.profile.coins", viewer = player, "coins" to "1,250")
        }
    }

fun profileTablist(player: Player) =
    tablist {
        headerLang("tablist.profile.header", viewer = player, "player" to player.name)
        footerLang("tablist.profile.footer", viewer = player, "coins" to "1,250")
    }
```

## Step 4: connect everything in one command group

```kotlin
@DaisyCommandSet
object ProfileCommands : DaisyCommandGroup({
    command("profile") {
        description("Open your profile tools")

        player {
            player.openMenu(lang("menus.profile.title"), rows = 3) {
                background(Material.GRAY_STAINED_GLASS_PANE) {
                    name(" ")
                }

                slot(13) {
                    item(Material.PLAYER_HEAD) {
                        nameLang("menus.profile.card.name", viewer = player, "player" to player.name)
                        loreLang("menus.profile.card.lore", viewer = player, "player" to player.name)
                    }
                }
            }

            daisy.scoreboards?.show(player, profileSidebar(player))
            daisy.tablists?.show(player, profileTablist(player))
        }
    }
})
```

<div class="premium-callout current">
  <h3>Why this version is cleaner</h3>
  <p>The same installed text source now drives commands, menus, sidebars, and tablists. You do not need plugin-local lang wrappers for each system anymore.</p>
</div>

## Why this example matters

The point is not that DaisyCore forces these systems together. The point is that they can work together cleanly because they share one platform owner, one lifecycle model, and one general API style.

## What to do next in a real plugin

- move shared UI definitions into dedicated files
- replace hard-coded values with placeholder-backed renderers
- only enable the systems your plugin actually needs
