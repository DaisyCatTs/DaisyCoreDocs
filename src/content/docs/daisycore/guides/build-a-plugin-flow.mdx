# Build a Plugin Flow

This guide shows how the main DaisyCore systems fit together in one believable feature instead of disconnected snippets.

## What you'll build

A `/profile` flow where:

- `/profile` opens a menu
- the menu acts as the player's profile hub
- a sidebar mirrors a few player stats
- a tablist shows a simple welcome header and footer

## The important rule before you start

You can build this entire flow with direct MiniMessage strings.

Shared config-backed text is the scaling path for plugins that already keep text in config or lang files.
It is not the requirement for a normal modern DaisyCore feature.

## Step 1: bootstrap DaisyCore once

```kotlin
class ProfilePlugin : JavaPlugin() {
    lateinit var daisy: DaisyPlatform
        private set

    override fun onEnable() {
        daisy =
            DaisyPlatform.create(this) {
                commands()
                menus()
                scoreboards()
                tablists()
            }
    }

    override fun onDisable() {
        daisy.close()
    }
}
```

## Step 2: build sidebar and tablist with direct MiniMessage strings

```kotlin
fun profileSidebar(player: Player) =
    sidebar {
        title("<gradient:#7dd3fc:#c4b5fd>Profile</gradient>")
        line("coins") {
            text("<gray>Coins: <white>1,250</white>")
        }
    }

fun profileTablist(player: Player) =
    tablist {
        header("<gradient:#7dd3fc:#c4b5fd>Welcome back</gradient>\n<white>${player.name}</white>")
        footer("<gray>Coins: <white>1,250</white>")
    }
```

## Step 3: connect everything inside one command group

```kotlin
@DaisyCommandSet
object ProfileCommands : DaisyCommandGroup({
    command("profile") {
        description("Open your profile tools")

        player {
            reply("<gradient:#7dd3fc:#c4b5fd>Opening your profile.</gradient>")

            player.openMenu("Profile", rows = 3) {
                background(Material.GRAY_STAINED_GLASS_PANE) {
                    name(" ")
                }

                slot(13) {
                    item(Material.PLAYER_HEAD) {
                        name("<gradient:#7dd3fc:#c4b5fd>${player.name}</gradient>")
                        lore(
                            "<gray>Inline Daisy menu flow</gray>",
                            "<white>Built for ${player.name}</white>",
                        )
                    }

                    message("<gray>Clicked the profile card.</gray>")
                    closeOnClick()
                }
            }

            daisy.scoreboards?.show(player, profileSidebar(player))
            daisy.tablists?.show(player, profileTablist(player))
        }
    }
})
```

## Step 4: move to shared text only when the plugin is config-heavy

If your plugin keeps wording in config or lang files, install one shared text source and swap local strings for:

- `replyLang(...)`
- `lang(...)`
- `nameLang(...)`
- `titleLang(...)`
- `headerLang(...)`
- related text helpers

That is the right time for config paths like:

- `profile.opening`
- `profile.menu.title`
- `profile.sidebar.title`
- `profile.tablist.header`

Those are useful config paths, not DaisyCore requirements.

<div class="premium-callout current">
  <h3>Why this matters</h3>
  <p>The same DaisyCore systems support both styles: direct MiniMessage strings for fast feature work, and shared key-backed text for config-heavy plugins.</p>
</div>

## When to choose each style

- use direct MiniMessage strings for most feature-local UI and command feedback
- use components when you want explicit low-level Adventure control
- use shared config-backed text when your plugin already stores wording outside code
