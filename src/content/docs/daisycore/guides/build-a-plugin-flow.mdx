# Build a Plugin Flow

This guide shows how the main DaisyCore systems fit together in one believable feature instead of isolated toy snippets.

<div class="premium-callout note">
  <h3>What this page assumes</h3>
  <ul>
    <li>You already have a working DaisyCore bootstrap.</li>
    <li>You have read the basics of commands, menus, scoreboards, and tablists.</li>
  </ul>
</div>

## What you'll build

A `/profile` flow:

- `/profile` opens a menu
- the menu acts as the player's profile hub
- a sidebar mirrors a few player stats
- a tablist shows a simple welcome header and footer

## Important rule before you start

You can build this entire flow with plain strings and MiniMessage strings.

Shared key-backed text is the scaling path for plugins that already keep messages in config or lang files. It is not a requirement for small or local UI text.

## Step 1: bootstrap DaisyCore once

```kotlin
class ProfilePlugin : JavaPlugin() {
    lateinit var daisy: DaisyPlatform
        private set

    override fun onEnable() {
        daisy =
            DaisyPlatform.create(this) {
                commands()
                menus()
                scoreboards()
                tablists()
            }
    }

    override fun onDisable() {
        daisy.close()
    }
}
```

## Step 2: build the flow with plain strings first

```kotlin
fun profileSidebar(player: Player) =
    sidebar {
        title("<gradient:#7dd3fc:#c4b5fd>Profile</gradient>")
        line("coins") {
            text("<gray>Coins: <white>1,250</white>")
        }
    }

fun profileTablist(player: Player) =
    tablist {
        header("<gradient:#7dd3fc:#c4b5fd>Welcome back</gradient>\n<white>${player.name}</white>")
        footer("<gray>Coins: <white>1,250</white>")
    }
```

## Step 3: connect everything in one command group

```kotlin
@DaisyCommandSet
object ProfileCommands : DaisyCommandGroup({
    command("profile") {
        description("Open your profile tools")

        player {
            replyMm("<gradient:#7dd3fc:#c4b5fd>Opening your profile.</gradient>")

            player.openMenu("Profile", rows = 3) {
                background(Material.GRAY_STAINED_GLASS_PANE) {
                    name(" ")
                }

                slot(13) {
                    item(Material.PLAYER_HEAD) {
                        name("<gradient:#7dd3fc:#c4b5fd>${player.name}</gradient>")
                        lore(
                            "<gray>Inline Daisy menu flow",
                            "<white>Built for ${player.name}",
                        )
                    }

                    messageMm("<gray>Clicked the profile card.</gray>")
                    closeOnClick()
                }
            }

            daisy.scoreboards?.show(player, profileSidebar(player))
            daisy.tablists?.show(player, profileTablist(player))
        }
    }
})
```

## Step 4: switch to shared text only when your plugin is config-heavy

If your plugin keeps titles, lore, sidebars, and tablist text in config or lang files, install one shared `DaisyTextSource` and swap the local strings for `lang(...)`, `replyLang(...)`, `nameLang(...)`, `titleLang(...)`, `headerLang(...)`, and the related helpers.

That is when example keys like:

- `messages.profile.opening`
- `menus.profile.title`
- `sidebar.profile.title`
- `tablist.profile.header`

become useful. They are config paths, not required Daisy naming.

<div class="premium-callout current">
  <h3>Why this matters</h3>
  <p>The same DaisyCore systems support both styles: local MiniMessage strings for quick feature code, and shared key-backed text for config-heavy plugins.</p>
</div>

## When to choose each style

- use local strings for fast feature work and small plugins
- use MiniMessage strings when you want rich formatting inline
- use shared text keys when your plugin already stores text in config or lang files
- use `Component` values when your plugin is already component-first
