# Config and Lang Text

This guide shows the intended DaisyCore workflow for config-heavy plugins.

But first, the important rule:

<div class="premium-callout current">
  <h3>Start simple</h3>
  <p>You do not need keys for basic messages. DaisyCore already supports plain strings, MiniMessage strings, and components. Shared text keys are the clean scaling path for plugins that already store text in config or lang files.</p>
</div>

## What to use when

- use `reply("...")` for plain command messages
- use `replyMm("...")` for MiniMessage command messages
- use plain strings in menu, sidebar, and tablist builders when the text is local
- use shared lang keys when the text lives outside code

## Step 1: install a shared text source

```kotlin
object PluginText : DaisyTextSource {
    override fun text(key: String): String? = pluginConfig.getString(key)

    override fun textList(key: String): List<String> = pluginConfig.getStringList(key)
}

class MyPlugin : JavaPlugin() {
    private lateinit var daisy: DaisyPlatform

    override fun onEnable() {
        daisy =
            DaisyPlatform.create(this) {
                messages(PluginText)
                commands()
                menus()
                scoreboards()
                tablists()
            }
    }
}
```

That object is just the adapter between your plugin config and DaisyCore. If you already keep text in YAML, JSON, TOML, or some lang system, this is where you connect it.

## Step 2: use plain strings and MiniMessage when the text is local

```kotlin
command("profile") {
    player {
        reply("Opening your profile.")
        replyMm("<gradient:#7dd3fc:#c4b5fd>Ready.</gradient>")
    }
}

player.openMenu("Profile", rows = 3) {
    slot(13) {
        item(Material.PLAYER_HEAD) {
            name("<gradient:#7dd3fc:#c4b5fd>${player.name}</gradient>")
        }
    }
}

sidebar {
    title("<gradient:#7dd3fc:#c4b5fd>Profile</gradient>")
}

tablist {
    header("<gradient:#7dd3fc:#c4b5fd>Welcome back</gradient>")
}
```

## Step 3: use shared keys when your plugin text already lives in config or lang files

```kotlin
@DaisyCommandSet
object ProfileCommands : DaisyCommandGroup({
    command("profile") {
        player {
            replyLang("messages.profile.opening", "player" to player.name)
        }

        subcommand("sync") {
            player {
                loading()

                profileService.sync(player.uniqueId).replyResult(
                    context = this,
                    success = {
                        replyLang("messages.profile.synced", "player" to player.name)
                    },
                    failure = { error ->
                        failLang("messages.profile.error", "error" to error.message.orEmpty())
                    },
                )
            }
        }
    }
})
```

```kotlin
player.openMenu(lang("menus.profile.title"), rows = 3) {
    background(Material.GRAY_STAINED_GLASS_PANE) {
        name(" ")
    }

    slot(13) {
        item(Material.PLAYER_HEAD) {
            nameLang("menus.profile.card.name", viewer = player, "player" to player.name)
            loreLang("menus.profile.card.lore", viewer = player, "player" to player.name)
        }

        messageLang("messages.profile.clicked", "player" to player.name)
        closeOnClick()
    }
}
```

## What DaisyCore handles for you

- key lookup through the installed `DaisyTextSource`
- placeholder replacement with `Pair<String, Any?>`
- MiniMessage parsing
- viewer-aware PlaceholderAPI rendering when a player is available
- shared text style across commands, menus, sidebars, and tablists

## What those keys actually are

Keys like:

- `messages.profile.opening`
- `menus.profile.title`
- `sidebar.profile.title`
- `tablist.profile.header`

are only example config paths.

They are not a DaisyCore requirement. They only make sense if your plugin already stores text by key. If your plugin does not, keep using local strings or components.

## When DaisySeries belongs in the same plugin

If your text flow also needs to parse materials, sounds, item flags, enchantments, or potion effects from config-heavy input, add [DaisySeries](/daisyseries/) beside DaisyCore instead of rebuilding parsing helpers locally.
