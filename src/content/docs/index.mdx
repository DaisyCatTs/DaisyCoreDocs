# DaisyCore

<div class="home-hero">
  <span class="home-kicker">One Mega Paper Library</span>
  <h2>The premium Kotlin-first toolkit for Paper plugins that want one dependency, one platform owner, and zero library sprawl.</h2>
  <p>DaisyCore brings commands, menus, sidebars, tablists, placeholders, items, and runtime ownership together so your plugin stack feels coherent instead of stitched together.</p>

```kotlin
dependencies {
    implementation("cat.daisy:DaisyCore:1.0.0")
}
```

## What DaisyCore includes

<div class="home-grid">
  <div class="home-card">
    <h3>Commands</h3>
    <p>Auto-loaded providers, Kotlin DSL builders, and one command bootstrap path through <code>DaisyPlatform</code>.</p>
  </div>
  <div class="home-card">
    <h3>Menus</h3>
    <p>Immutable menu definitions with session-based runtime handling, inventory safety, and refresh support.</p>
  </div>
  <div class="home-card">
    <h3>Scoreboards</h3>
    <p>Sidebar-first sessions with keyed lines, invalidate-by-key updates, and clean platform ownership.</p>
  </div>
  <div class="home-card">
    <h3>Tablists</h3>
    <p>Header and footer rendering with per-player sessions and a simple baseline API that stays honest.</p>
  </div>
  <div class="home-card">
    <h3>Placeholders</h3>
    <p>A shared placeholder registry so menus, sidebars, tablists, and messages can resolve viewer-aware values consistently.</p>
  </div>
  <div class="home-card">
    <h3>Items</h3>
    <p>Reusable item builders that keep Adventure text and inventory rendering code cleaner across the platform.</p>
  </div>
</div>

## Why DaisyCore

<div class="home-panel">
  <h2>One dependency, shared foundations, cleaner plugin code</h2>
  <p>DaisyCore is designed so normal plugin projects depend on one thing. Internally, the repo stays modular. Publicly, your plugin code gets one bootstrap path, one text model, one placeholder system, and one family of Daisy-branded types.</p>
</div>

## Current status

<div class="api-note">
  <p><strong>Current command discovery:</strong> DaisyCore currently scans your plugin jar for <code>@DaisyCommandSet</code> providers at startup.</p>
  <p><strong>Scoreboards and tablists:</strong> both are production baseline systems. They are meant to be usable now, while leaving room for deeper transports later.</p>
</div>
