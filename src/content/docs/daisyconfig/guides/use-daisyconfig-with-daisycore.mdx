# Use DaisyConfig with DaisyCore

Use DaisyCore for:

- commands
- menus
- sidebars
- tablists
- runtime ownership
- MiniMessage rendering

Use DaisyConfig for:

- typed YAML values
- reload-safe config ownership
- config-backed text

## The important rule

DaisyConfig stores and loads text.
DaisyCore renders it.

That means a clean integration looks like this:

```kotlin
val langHandle = yamlTextConfigHandle(plugin, "lang.yml")

val daisy =
    DaisyPlatform.create(plugin) {
        messages(langHandle.current.asDaisyTextSource())
        commands()
        menus()
        scoreboards()
        tablists()
    }
```

## What DaisyCore still owns

Even when the text came from config, DaisyCore still owns:

- MiniMessage parsing
- viewer-aware rendering
- PlaceholderAPI integration
- fallback behavior when PlaceholderAPI is absent

## What DaisyConfig does not do

DaisyConfig does **not**:

- expand PlaceholderAPI directly
- become a second rendering runtime
- replace DaisyCore's UI/render model

## Normal production pattern

Use:

- direct MiniMessage strings for feature-local text
- DaisyConfig-backed text when you want shared, reloadable, or admin-editable wording

That gives you the best of both models without mixing responsibilities.

## Example

```kotlin
val profileHandle = yamlConfigHandle("profile-ui.yml", profileUiCodec)
val langHandle = yamlTextConfigHandle(plugin, "lang.yml")

val daisy =
    DaisyPlatform.create(plugin) {
        messages(langHandle.current.asDaisyTextSource())
        commands()
        menus()
        scoreboards()
        tablists()
    }

val sidebar =
    sidebar {
        title(profileHandle.current.sidebarTitle)
        line("coins") {
            textLang("profile.sidebar.coins", viewer = player, "coins" to "1,250")
        }
    }
```
