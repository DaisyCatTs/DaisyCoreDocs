# DaisyConfig

<div class="home-hero">
  <span class="home-kicker">Typed YAML For Paper</span>
  <h2>Use DaisyConfig when config loading, validation, reload safety, and config-backed text are the part you are tired of rebuilding.</h2>
  <p>DaisyConfig is the typed config layer in the Daisy ecosystem. It is YAML-first, codec-driven, reload-safe, DaisySeries-aware, and designed to hand clean config-backed text into DaisyCore without creating a second rendering runtime.</p>

  <div class="home-actions">
    <a class="home-button" href="/daisyconfig/getting-started/install/">Install DaisyConfig</a>
    <a class="home-button-secondary" href="/daisyconfig/getting-started/first-config/">Build your first typed config</a>
  </div>
</div>

## What DaisyConfig gives you

<div class="feature-matrix">
  <div class="feature-card"><strong>Typed config mapping</strong><p>Replace `getString(...)` sprawl with explicit codecs and real config objects.</p></div>
  <div class="feature-card"><strong>Reload-safe handles</strong><p>Keep the last valid config in memory if a reload fails.</p></div>
  <div class="feature-card"><strong>DaisySeries-aware codecs</strong><p>Decode materials, sounds, item flags, enchantments, and potion effects directly from config-friendly strings.</p></div>
  <div class="feature-card"><strong>DaisyCore text bridge</strong><p>Provide config-backed text to menus, sidebars, tablists, and command replies without bypassing DaisyCore’s render model.</p></div>
</div>

## Important safety rule

<div class="premium-callout current">
  <h3>DaisyConfig does not expand PlaceholderAPI directly</h3>
  <p>DaisyConfig owns typed YAML and text-source data. DaisyCore still owns MiniMessage parsing, viewer-aware rendering, PlaceholderAPI integration, and fallback behavior when PlaceholderAPI is absent.</p>
</div>
