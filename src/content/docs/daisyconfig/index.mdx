## What DaisyConfig handles

<div class="feature-matrix">
  <div class="feature-card"><strong>Codec-driven config</strong><p>Map YAML into real objects with explicit shape, path-aware validation, and DaisySeries-backed value codecs.</p></div>
  <div class="feature-card"><strong>Unmanaged YAML</strong><p>Keep the low-level `DaisyYaml` and simple handle path when one file and one codec are enough.</p></div>
  <div class="feature-card"><strong>Managed YAML</strong><p>Own `config_version`, default-file creation, missing-key merge, ordered migrations, and last-good runtime preservation.</p></div>
  <div class="feature-card"><strong>Module bundles</strong><p>Load `modules/&lt;category&gt;/&lt;module&gt;/settings.yml` plus `lang.yml` as one typed settings-and-text unit.</p></div>
</div>

## What DaisyConfig does not handle

<div class="feature-matrix">
  <div class="feature-card"><strong>No placeholder execution</strong><p>DaisyConfig stores text and config data only. It does not call PlaceholderAPI or run a second render pipeline.</p></div>
  <div class="feature-card"><strong>No viewer-aware rendering</strong><p>DaisyCore still owns MiniMessage parsing, viewer-aware rendering, and runtime UI behavior.</p></div>
  <div class="feature-card"><strong>No parser ownership</strong><p>DaisySeries still owns canonical keys, aliases, and value parsing rules for the broader modern Paper value surface, including materials, sounds, biomes, attributes, difficulties, particles, and more.</p></div>
</div>
